# Lab Practice 1 — Arithmetic Calculator using SLY

An Arithmetic calculator that reads an arithmetic expression as text and evaluates it directly to a number
— e.g. `2+3*5` → `17`. Files: `CalcLexer.py`, `CalcParser.py`, `calculator.txt` (problem
statement).

This is the first hands-on use of SLY (Sly Lex Yacc) — two classes Lexer, Parser are used to build lexical analyzer, syntax analyzer(parser); everything in this course builds on this same pair of classes.


## SLY Lexer, briefly

A **lexer** performs lexical analysis. It reads the input expression and breaks it into a sequence of **tokens**.

For example, the expression:

```text
10 + 20 * 5
```

is converted into tokens such as:

```text
NUMBER(10)  '+'  NUMBER(20)  '*'  NUMBER(5)
```

The lexer specification is defined by creating a class that inherits from `sly.Lexer`.

`CalcLexer.py` has the following lexer specification in CalcLexer class 

- **`literals`** — `{'+', '-', '*', '/', '%'}`. For single-character symbols that don't need
  their own named token type, SLY lets you list them as `literals` instead of writing a regex
  rule for each. The grammar can then refer to them directly as quoted characters
  (`@_('E "+" T')`), which is what `CalcParser.py` does.
- **`NUMBER` as a function-style rule.** Written as a method rather than a plain string
  attribute specifically so it can convert the matched text to an actual Python `int`
  (`t.value = int(t.value)`) before the token reaches the parser — a plain string rule can only
  assign a token type, not transform the matched value.

Run the lexer standalone (uncommented at the bottom of `CalcLexer.py`) to see the token stream
for `2+3*5` before constructing parser.

## SLY Parser, briefly

A **parser** takes the tokens generated by the lexer and checks whether they follow the grammar of the language.

The parser specification is defined by creating a class that inherits from `sly.Parser`.

`CalcParser.py` has the synatx specification pf `CalcParser` class

`CalcParser` consumes the token stream and evaluates it, using the classical grammar from
`calculator.txt`:

```
E -> E + T | E - T | T
T -> T * F | T / F | T % F | F
F -> NUMBER
```

Note what's **not** here: there's no `precedence` table. This grammar is written in three
layers (`E`, `T`, `F`) specifically so that precedence and associativity fall out of the
grammar's *shape* — `T` (terms: `*`, `/`, `%`) only ever appears nested inside `E` (sums:
`+`, `-`), so multiplication/division/remainder are structurally forced to bind tighter than
addition/subtraction, with no separate precedence declaration needed. This is an alternative to
the flat-grammar-plus-`precedence`-table style you'll use starting Week 3 for TinyCStr — both
approaches are standard, and comparing them is worth a moment's thought: the `E`/`T`/`F` style
scales awkwardly as more operators/precedence levels are added (one more grammar layer per
level), which is exactly why TinyCStr's own grammar uses the flatter `precedence`-table style
instead once expressions get more complex.

Each grammar rule here directly computes and returns a number (`value[0] + value[2]`, etc.) —


## Running it

```bash
python3 CalcParser.py
```
prints `20` for the hardcoded `inp = '10-2*3+2*5'` at the bottom of the file. Edit `inp` to try
other expressions, or import `CalcLexer`/`CalcParser` from your own script.
